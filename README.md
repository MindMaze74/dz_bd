### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

- Для бюджетирования и аналитики лучше всего подойдёт реляционная СУБД например, PostgreSQL, Oracle или Microsoft SQL Server. Эти системы хорошо держат структуру данных, не дают им «развалиться», поддерживают сложные запросы для отчётов и гарантируют, что все операции либо выполняются целиком, либо откатываются (это как раз про надёжность финансовых расчётов).

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

- Если хеширование тормозит, сначала тестируется встроенные функции СУБД. Если этого мало, подключаются готовые библиотеки: OpenSSL или Intel IPP для криптографических задач, а для некриптографических целей Google CityHash или xxHash. В крайнем случае можно задействовать аппаратное ускорение (специальные инструкции процессора или модули безопасности), но это уже для особо нагруженных систем.

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

- Для лендингов удобнее NoSQL базы вроде MongoDB или Couchbase: они позволяют быстро менять структуру данных и хранить заявки в формате JSON. Для CRM лучше реляционная СУБД (PostgreSQL или MySQL) там важны связи между клиентами, сделками и контактами, а также целостность данных.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

- Да можно обойтись одной PostgreSQL. В ней есть тип данных JSONB он подходит для хранения заявок с лендингов без жёсткой схемы. А для CRM сущностей (контакты, сделки) используем обычные таблицы со связями. Так мы получаем и гибкость, и надёжность в одной системе.

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

Для базы знаний подойдут два варианта:

- MongoDB если нужно просто хранить статьи, инструкции, PDF и видео в виде документов с метаданными;

- Neo4j если важно показать связи между правилами (например, «правило А ссылается на правило Б»).

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

- Да, можно использовать ту же PostgreSQL из задачи 1.2. Создается отдельная схема knowledge_base, где будут таблицы для отделов и категорий, а сами документы хранятся в поле JSONB. Так не будем заводить новую систему, а централизуем данные.

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

Для логистики идеальна графовая СУБД Neo4j. Она быстро считает маршруты и распределяет курьеров, потому что разбирает связи между объектами, дорогами и точками доставки лучше, чем обычные таблицы.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

Лучше разделить: 

- Neo4j для логистики (маршруты)

- PostgreSQL для закупок (поставщики, номенклатура)

Связываются они через API или сервисную шину. Так каждая система останется быстрой и понятной, а данные не перемешаются.

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

- Теоретически всё можно уместить в PostgreSQL, но это рискованно: система станет сложной, медленной и уязвимой к сбоям.
На практике лучше гибридный подход: PostgreSQL для финансов, CRM и базы знаний, Neo4j для логистики. 
Связка их через API будет надёжнее, быстрее и проще поддерживать.Это даст лучшую производительность, масштабируемость и снизит риски единой точки отказа.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

1. Пользователь заходит в приложение, выбирает пополнение, вводит номер и сумму, подтверждает.

2. Система проверяет номер (чтобы он был правильный), авторизует пользователя и смотрит, хватает ли денег на счёте.

3. Если всё ок, сумма временно блокируется чтобы её случайно не потратили в другом месте.

4. Система отправляет запрос оператору связи «пополнить номер на столько‑то».

5. Оператор подтверждает, что всё прошло, баланс обновился. Система списывает деньги и записывает транзакцию.

6. Пользователь получает уведомление (в приложении или SMS), операция завершена.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

1. Пользователь заранее настраивает в приложении номер, сумму и условие (например, «пополнить, если баланс ниже 100 руб.» или «каждый месяц 5‑го числа»).

2. Система следит за балансом или датой.

3. Когда условие выполняется, автоматически запускается пополнение.

4. Проверяется, хватает ли денег. Если нет процесс прерывается, пользователь получает уведомление об ошибке.

5. Если деньги есть, система блокирует сумму, отправляет запрос оператору, списывает средства и фиксирует операцию.

6. Пользователь получает уведомление, а система готовится к следующему циклу проверки.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

- SQL системы обеспечивают строгую целостность данных через поддержку ACID транзакций, имеют чётко заданную схему данных (фиксированную структуру таблиц), используют единый стандартизированный язык запросов (SQL), располагают обширной экосистемой инструментов и библиотек для разработки и администрирования, а также отличаются высокой степенью безопасности и развитыми механизмами контроля доступа.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

- NewSQL системы сочетают поддержку ACID согласованности (как в SQL) с высокой масштабируемостью (как в NoSQL), обеспечивают высокую производительность при обработке большого числа коротких транзакций, совместимы с языком SQL (сохраняя удобство для разработчиков), позволяют горизонтально масштабироваться без потери целостности данных и оптимизированы для работы с большими объёмами данных в распределённых средах.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

Критерий выбора СУБД: нужно, чтобы система могла расти вместе с объёмом данных (горизонтально масштабироваться), быстро считать на 1000 машин, не падать при сбоях отдельных серверов и поддерживать распределённую обработку. Важно понимать, что именно мы делаем с данными: анализируем (OLAP) или обрабатываем транзакции (OLTP) от этого зависит выбор движка.

Оптимальная модель распределённых вычислений: Apache Spark с Hadoop HDFS либо колоночная СУБД (ClickHouse/Greenplum). Spark обеспечит быструю обработку в памяти на кластере из 1 000 машин, Hadoop HDFS надёжное хранение с репликацией, а ClickHouse/Greenplum ускоренную аналитику за счёт колоночного хранения, распараллеливания запросов и сжатия данных. Такое решение гарантирует высокую скорость вычислений, устойчивость к сбоям и полное задействование выделенной инфраструктуры.
